# Git 提交与文档规范

适用于所有项目。

## Git 提交

- 一个功能 = 一个提交（自包含）
- 50/70 格式：标题 ≤50 字符，正文每行 ≤70 字符
- 标题使用英文动词开头 + 简要描述
- **禁止**提交 `.env`、API Key、密钥等敏感信息
- 提交前确认 `npm run lint` 和 `npm test` 通过

## 文档规范

- 技术方案 / 架构图放 `docs/` 目录，使用中文 Markdown
- 图源即文档（Mermaid），不另存 png / drawio
- 新增模块先改方案文档，再编码
- Provider / 模块特殊逻辑写对应的 `.spec.md` 测试文档

## 反合理化

| Agent 常见借口 | 为什么不行 |
|---|---|
| "先提交代码，文档后面补" | 方案文档先行，代码是方案的实现 |
| "这个提交包含多个功能没关系" | 一个提交一个功能，便于 review 和回滚 |

## 验证关卡

- [ ] 提交信息符合 50/70 格式
- [ ] 提交中无敏感信息
- [ ] 涉及架构变更时 `docs/` 已更新
