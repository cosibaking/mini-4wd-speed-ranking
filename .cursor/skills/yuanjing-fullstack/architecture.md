# 架构设计约束

适用于所有服务。部署模式选型见 [infra.md](infra.md)。

## 多实例部署（K8s 服务）

- K8s 服务以 Deployment 部署，Pod 数量随负载动态调整
- **所有方案设计必须确保 N 个实例并行运行时行为正确**——最高优先级约束
- 编写新功能时先自问："如果同时有 3 个实例在运行，这段代码会不会出问题？"

## 进程无状态

- **禁止**在进程内存中持有跨请求的可变业务状态（缓存用户数据、请求计数器、任务进度等）
- 启动时初始化的固定映射（如配置、路由表）属于"配置态"，可以保留
- 需要跨请求或跨 Pod 共享的可变状态，**必须存入 Redis 或数据库**
- 本地文件系统不可靠（Pod 可能随时被驱逐重建），**禁止**写入业务数据

## 分布式协调走中间件

- 限流、并发控制、熔断器状态、队列——一律通过 Redis 等中间件
- 如需分布式锁，使用 Redis（**禁止**进程内 mutex / 内存锁，多 Pod 场景无效）
- 新增"全局唯一"或"全局计数"需求时，必须走原子操作

## 幂等性

- 消息队列的 Job 可能因 Pod 重启被重新投递，Processor 必须能安全地重复执行
- Webhook 回调使用唯一 ID 去重
- 多表关联写入使用数据库事务保证原子性
- 设计新功能时明确回答："如果这个操作执行了两次，结果是否一致？"

## 优雅关停

- **K8s 服务**：进程收到 SIGTERM 后应停止接收新请求、等待进行中请求完成、释放资源
- **FC 服务**：函数执行完毕即释放，不依赖长生命周期资源
- 新增长生命周期资源（连接池、定时器、EventEmitter、流）时，**必须**在 shutdown 钩子中释放
- 避免在 shutdown 流程中发起新的异步操作

## 健康检查（K8s 服务）

- K8s 就绪/存活探针必须快速响应（< 1s）
- 新增外部依赖时评估是否纳入健康检查
- **禁止**在探针中执行耗时查询或外部调用
- FC 服务无需自定义探针，由平台管理

## 服务边界

- 每个服务只做自己领域内的事，不承载其他服务的业务逻辑
- 服务间通过 API 交互，**不直连**对方的 DB / Redis——双向隔离
- 配置、路由规则、权限全部 **DB 驱动**，不硬编码在代码或配置文件中

## 向后兼容

- **gRPC proto**：不删字段、不改字段号、新增字段用新编号
- **HTTP API**：路径变更保持旧路径可用，或通过版本前缀区分
- **数据库 Schema**：变更通过 Migration，**禁止**手动 DDL
- **消息/Job 数据**：变更结构时确保已在队列中的旧格式仍可正确处理

## 前端部署约束

- SSR 服务同样以多实例部署，渲染逻辑必须**无状态**（不依赖进程内缓存）
- 静态资源走 CDN，构建产物使用内容哈希命名确保缓存安全
- 前后端通过 API 契约交互，前端不直连数据库
- 环境变量与密钥管理见 [nextjs.md](nextjs.md) 和 [security.md](security.md)

## 反合理化

| Agent 常见借口 | 为什么不行 |
|---|---|
| "这个功能只在单实例跑，不需要考虑多 Pod" | 多实例是 Day 1 约束，所有代码默认在 N 个 Pod 并行 |
| "先用内存缓存，后面再改 Redis" | 上线就是多实例，内存缓存会导致数据不一致 |
| "这个小改动不影响兼容性" | 正在运行的旧 Pod / 队列中的旧 Job 可能还在用旧格式 |
| "shutdown 不需要清理，Pod 重启就行" | 不清理会导致连接泄漏、数据不一致、队列 Job 丢失 |

## 验证关卡

- [ ] 新增功能没有在进程内存中持有可变业务状态
- [ ] 需要共享的状态已通过 Redis / DB 管理
- [ ] 操作是幂等的（重复执行结果一致）
- [ ] 新资源在 shutdown 钩子中正确释放
- [ ] API / Schema / 消息格式变更向后兼容
